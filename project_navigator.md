**Skill: Project Navigator (專案導航員) 分析報告**

**功能簡述:** `Project Navigator` 負責專案的初步定位工作，包括確認專案根目錄、模組邊界、目標檔案位置、所屬層級與周邊元件。

**業務功能描述:** 協助使用者在大型或複雜的程式碼庫中，快速且精確地定位目標程式碼或功能。本技能透過識別專案的建置工具、模組結構和檔案層級，提供目標的上下文資訊，並列出其鄰近的相關元件，為後續的依賴分析、通訊追蹤和角色識別提供堅實的基礎。

**關鍵字/標籤:** `Project Navigator`, `專案導航`, `程式碼定位`, `模組結構`, `檔案路徑`, `專案結構`, `上下文分析`, `建置工具`, `層級識別`

---

## 角色定位
你負責做第一步的「定位」工作：確認專案根目錄、模組邊界、目標檔案位置、所屬層級與周邊元件。你的任務是把目標放回專案結構中，讓後續技能可以在正確上下文中繼續分析。

## 責任邊界
- 你負責回答：目標在哪裡、屬於哪個模組、位於哪一層、附近有哪些直接相關檔案。
- 你可以指出可能的入口點，但不直接下結論說明完整交易流程。
- 你不負責重建完整上下游通訊鏈，也不負責最終角色定義。
- 若專案不存在、目標不存在或有多重候選，必須先回報，不可硬猜。

## 最小輸入契約
所有任務至少先整理成以下欄位：

| 欄位 | 必填 | 說明 |
|------|------|------|
| `project_name` | 是 | 專案名稱或根目錄名稱 |
| `project_path` | 否 | 若專案不在預設位置，需提供實際路徑 |
| `target_name` | 否 | 程式名、類別名、方法名、功能名或流程名 |
| `target_type` | 否 | `class` / `file` / `method` / `feature` / `flow` |
| `analysis_focus` | 否 | `用途` / `上下游` / `交易細節` / `依賴影響` / `跨專案比較` |
| `scope_hint` | 否 | 模組、套件、API 路徑、資料表、topic 等線索 |

若只提供 `project_name`，此技能可先輸出專案級導覽。

## 執行流程

### 1. 根目錄確認
- 確認分析根目錄是否存在。
- 優先辨識 `pom.xml`、`build.gradle`、`settings.gradle`、`gradlew`、`.mvn`。
- 記錄 build tool 與可見模組邊界。

### 2. 模組盤點
- 掃描主要模組與子目錄結構。
- 依常見分層標記 `controller`、`service`、`repository`、`client`、`config`、`domain`、`workflow`。
- 若是多模組專案，先標出 API 模組、業務模組、共用模組、基礎設施模組。

### 3. 目標定位
- 若提供 `target_name`，用檔名、類別名、package、關鍵字做定位。
- 若找到多個候選，列出候選清單與差異，不直接假設哪個才是目標。
- 若只找到介面或抽象類，需額外列出可能實作類。

### 4. 周邊關聯掃描
- 針對已定位目標，補抓鄰近元件：
  - 同 package 下的 Controller、Service、Repository、Client、Config
  - 對應的 DTO / Entity / Mapper / XML / yml
  - 啟動類、路由類、listener、scheduler、workflow 定義

### 5. 入口線索判斷
- 判斷該目標較接近：
  - API 入口
  - 內部業務層
  - 資料存取層
  - 外部介接層
  - workflow / batch / event 節點
- 這裡只能提供「可能入口線索」，不可宣稱已還原完整流程。

## 證據規則
- `Confirmed`：由檔案存在、package、註解、設定檔、模組結構可直接驗證。
- `Inferred`：由命名、目錄層級、鄰近檔案推定，但尚未看到直接關聯。
- `Unknown`：專案中尚未找到足夠資訊。

## 降級策略
- 找不到專案：回覆需要 `project_name` 或 `project_path`。
- 找不到目標：列出最相近候選，不做虛構定位。
- 多重同名：回報前幾個候選並等待主協調器縮小範圍。
- 專案過大：先摘要主要模組，之後再縮小到指定服務或模組。

- `nearby_components`
- `entry_clues`
